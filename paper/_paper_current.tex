\begin{abstract}
  Modern data landscapes increasingly rely on heterogeneous database systems spanning relational, document, graph, and wide-column paradigms.
  As application requirements change, schemas frequently require migration across systems or evolution within them.
  Existing schema evolution languages are mostly limited to intra-system scenarios, and no unified formal grammar currently supports schema migration and evolution across heterogeneous database systems.
  We present SMILE (Schema Modification and Evolution Language), a domain-specific language that addresses this gap.
  SMILE builds on the metamodel and adopts a Hub-and-Spoke architecture to reduce inter-system mapping complexity.
  The language defines a comprehensive operation taxonomy covering schema structure, constraints, and paradigm-specific operations, formalized through an ANTLR4-based grammar.
  We validate SMILE across all 16 migration and evolution scenarios, with 32/32 test configurations passing a two-layer validation pipeline.
  All transformations succeed with full structural correctness, with minor notices arising only from inherent nullability differences across database paradigms.
\end{abstract}

\keywords{Schema Migration, Schema Evolution, Domain-Specific Language, NoSQL, Metamodel, ANTLR4}

\section{Introduction}
\label{sec:introduction}
% ============================================================

As the volume and variety of data continue to grow, diverse SQL and NoSQL database systems are increasingly adopted to address
different application requirements. For instance, Sega moved from an RDBMS to MongoDB to improve performance~\cite{bansal2021journey}. SMILE
targets four database paradigms: relational, document, graph, and wide-column. These paradigms differ substantially in how they express relationships, since relational databases use foreign keys, document stores embed related data as sub-objects, and graph databases model associations as directed edges. Wide-column systems take a different approach altogether, denormalizing related data directly into columns.

\emph{Schema evolution} refers to modifying a schema within the same database system (intra-system)~\cite{brahmia2024literature}, while
\emph{schema migration} refers to transforming a schema between different database systems (inter-system)~\cite{googlecloud2024heterogeneous}.
Considering four paradigms, the combined space comprises 4 evolution scenarios (diagonal) and 12 migration scenarios, yielding 16 total scenarios
(Table~\ref{tab:migration-matrix}).

Existing schema evolution languages are mostly limited to intra-system scenarios~\cite{storl2022darwin, chillon2022taxonomy, curino2008graceful,
herrmann2015codel, herrmann2017living, hausler2023language}, and inter-system migration work~\cite{meier2023datenmigration} has not been
consolidated with evolution operations into a unified formal grammar.

To address this gap, we present \textbf{SMILE}, a domain-specific language that supports both schema evolution
(intra-system) and schema migration (inter-system) within a single ANTLR4-based grammar.
SMILE integrates the evolution operations defined in the Orion taxonomy~\cite{chillon2022taxonomy} with the structural transformation operations
from prior work within our research group, extending them with additional key and constraint operations required for relational and
wide-column systems.

To validate SMILE, we provide a prototypical implementation referred to as the \textbf{SMILE Engine}, built on the M-Model~\cite{conrad2022metamodels}
as its architectural foundation.
As shown in Figure~\ref{fig:SMILE-overview}, the engine adopts a Hub-and-Spoke architecture with adapters for PostgreSQL, MongoDB, Neo4j,
and Cassandra.
Source schemas are first reverse-engineered (RE) into an adapted meta model, which is then transformed by SMILE scripts and
forward-engineered (FE) into the target paradigm.
This design reduces the mapping complexity from $O(N^2)$ point-to-point translations to $O(N)$ adapter pairs, enabling a single language to
cover all 16 scenarios.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=\columnwidth]{{overview.drawio}.pdf}
  \caption{Overview of the SMILE language and engine.}
  \label{fig:SMILE-overview}
\end{figure}

To address these challenges, this paper contributes:
\begin{compactitem}
    \item A \textbf{comprehensive operation taxonomy} that consolidates evolution and migration operations into a complete set of atomic schema
change primitives.
    \item A \textbf{formal ANTLR4-based grammar (SMILE)} that implements this taxonomy as a domain-specific language covering all 16 migration
and evolution scenarios across four database paradigms.
    \item An \textbf{adapted meta model (M-Model\textsuperscript{+})} that extends the M-Model~\cite{conrad2022metamodels} to serve as the
canonical hub for cross-paradigm transformations.
    \item A \textbf{prototypical implementation (SMILE Engine)} with a Hub-and-Spoke architecture, a two-layer validation framework, and a
web-based visualization interface, validated across 32 test configurations.
\end{compactitem}

\section{Related Work}
\label{sec:related-work}
% ============================================================

This section reviews prior work on schema evolution and migration that informs the design of SMILE.

\subsection{Relational Schema Evolution}\todo{Why only relational schema evolution and not schema evolution for different single stores? I think we should fuse relational and meta model evolution}
The origins of schema evolution research lie in the relational paradigm.
Model Management~2.0~\cite{bernstein2007model} proposed generic operators for manipulating schemas and mappings, establishing a theoretical foundation for formal schema transformation.
Building on this, PRISM~\cite{curino2008graceful} introduced Schema Modification Operators (SMOs) that not only define schema changes but also automatically rewrite legacy queries, and PRIMA~\cite{moon2009prima} further enabled temporal queries over databases with evolving schemas by archiving historical data under their original schema versions.
CoDEL~\cite{herrmann2015codel} augments PRISM to achieve relational completeness, and its successor BiDEL~\cite{herrmann2017living} adds bidirectional capabilities so that multiple schema versions can coexist with changes propagating both forward and backward.
\textit{These works addressed evolution within relational systems but did not consider cross-database migration, as NoSQL systems were beginning to emerge during this period.}


\subsection{Multi-Model Evolution and Migration}
As NoSQL databases gained adoption, schema evolution research expanded beyond the relational paradigm.
Scherzinger, Klettke, and St\"orl~\cite{scherzinger2013managing} laid the theoretical foundation by defining a declarative language with five
operations and distinguishing between eager and lazy migration strategies.
Holub\-ov\'a, Klettke, and St\"orl~\cite{holubova2019evolution} identified five key challenges in multi-model data modelling and outlined an EBNF
syntax for a corresponding evolution language.
Since existing NoSQL evolution proposals typically rely on simple data models that omit aggregation and structural variations,
Orion~\cite{chillon2022taxonomy}, grounded in U-Schema~\cite{candel2022unified}, addresses this limitation by organizing schema change operations
into five categories (schema type, feature, attribute, reference, aggregate) and implementing them as a platform-independent DSL.
Darwin~\cite{storl2022darwin} takes a different direction, providing a data platform for NoSQL schema evolution management with a REST API, schema
version tracking, and various migration strategies.
Hausler et al.~\cite{hausler2023language} target graph databases specifically, defining evolution operations in an EBNF-like grammar with
implementation in Neo4j.
Su\'arez-Otero et al.~\cite{suarez2025data} propose MoDEvo, a model-driven approach that ensures data integrity during schema evolution in
column family databases (Apache Cassandra) by automatically generating data migration scripts through model transformations.
While MoDEvo addresses the critical challenge of data migration accompanying schema changes, it focuses on intra-system evolution within a single
paradigm and does not target inter-system schema migration.
While these approaches focus on intra-system evolution, prior work from our research group addresses inter-system migration by introducing structural transformation operations (NEST, FLATTEN, LINKING, UNWIND) for schema restructuring across heterogeneous systems.
This prior work reuses the basic evolution operations defined by Scherzinger et al.~\cite{scherzinger2013managing} and Holub\-ov\'a et al.~\cite{holubova2019evolution}, while introducing four structural transformation operations with separate syntax definitions.
\textit{However, these two sets of operations have not been consolidated into a unified formal grammar.}

\subsection{Metamodel Foundations}
This canonical hub requires a central, platform-independent metamodel.
Several metamodels have been proposed for multi-model database systems~\cite{atzeni2020data, abdelhedi2017umltonosql, vanhove2017data, mali2020modeldrivenguide, zdepski2020pddm, candel2022unified, conrad2022metamodels}, of which U-Schema~\cite{candel2022unified} and the M-Model~\cite{conrad2022metamodels} are most relevant to this work.
Candel et al.~\cite{candel2022unified} propose U-Schema, a unified logical metamodel capable of representing schemas for relational, document,
graph, key-value, and columnar systems.
U-Schema introduces \texttt{StructuralVariation}s to capture the schema-on-read variability common in NoSQL databases. Its feature hierarchy classifies \texttt{Attribute} and \texttt{Aggregate} as structural features, \texttt{Key} and \texttt{Reference} as logical
features, but the two branches do not share a common superclass.

Conrad et al.~\cite{conrad2022metamodels} present the M-Model, a metamodel designed specifically for schema and data migration between
heterogeneous database systems, with the principle of keeping complexity as low as possible while covering all five database paradigms.
Unlike U-Schema, the M-Model defines \texttt{Relationship} as a common superclass for \texttt{Reference}, \texttt{Embedded}, and \texttt{Edge},
enabling conversions between relationship types through a uniform interface.
Its \texttt{Key} concept includes an explicit \texttt{keyType} attribute that distinguishes between partition and clustering keys, which is
necessary for wide-column stores such as Cassandra.
Table~\ref{tab:metamodel-comparison-rw} summarizes the key differences.
SMILE adopts the M-Model as its architectural foundation, since its unified relationship hierarchy simplifies conversions between paradigms while
the explicit key typing covers Cassandra's partition-clustering structure.

\subsection{Research Gap}
The reviewed approaches have collectively established a rich foundation for schema evolution and migration, ranging from relational evolution languages to NoSQL taxonomies, structural transformation operations, and metamodel design.
Building on these contributions, the following four gaps remain.

\begin{enumerate}
    \item The existing operation sets each address specific paradigms or scopes, but a consolidated taxonomy that unifies them into a complete set of atomic primitives has not yet been established.

    \item Existing grammars~\cite{scherzinger2013managing, holubova2019evolution, chillon2022taxonomy, hausler2023language} each target individual paradigms or intra-system scenarios. Prior work from our research group defines separate syntax rules for evolution and structural transformation operations, which have not yet been consolidated. As shown in Table~\ref{tab:migration-matrix}, the combined space comprises 16 scenarios, and a unified formal grammar that covers all of them is still missing.

    \item The M-Model~\cite{conrad2022metamodels} provides a metamodel with a unified relationship hierarchy and explicit key typing. To serve as the canonical hub for both schema evolution and schema migration within a single language, additional structural elements such as constraint handling and entity type classification are needed.

    \item Existing tools each cover parts of the problem space effectively. A prototypical implementation that combines a Hub-and-Spoke architecture with a clear migration pipeline, a validation framework, and a web-based visualization interface would enable rapid feedback during language development, allowing grammar and operation design to be iteratively refined based on concrete migration results across all 16 scenarios.
\end{enumerate}

\textbf{Dual Grammar Variants.}
SMILE provides two grammar variants for every operation.
The \emph{Specific} grammar uses dedicated keywords per operation
(e.g., \texttt{ADD\_ENTITY}, \texttt{RENAME\_ATTRIBUTE}),
making each statement self-explanatory at the cost of a larger
lexer token set.
The \emph{Generalized} grammar takes a different approach,
using a compositional verb--object syntax.
Each statement combines a verb with an object keyword
(e.g., \texttt{ADD ENTITY}, \texttt{RENAME ATTRIBUTE}).
This reduces the number of lexer tokens and makes the grammar
easier to extend, though at the cost of slightly reduced readability.
In practice, the Specific grammar requires 43 dedicated keywords, while the Generalized grammar achieves the same coverage with only 24 composable tokens (7~verbs, 6~object types, 5~key modifiers, and 6~structural operations), which reduces the lexer complexity by approximately 45\%.
Both variants share the same semantic operations and produce
identical results through the M-Model\textsuperscript{+}.

\textbf{Entity Operations.}
This category covers operations on entire entities.
\texttt{ADD} creates a new entity type, \texttt{DELETE} removes one, \texttt{RENAME} changes its name, and \texttt{COPY} duplicates its structure.
\texttt{MERGE} combines two entities into one, while \texttt{SPLIT} partitions an entity into two by attribute subsets.
Orion defines a separate \texttt{EXTRACT} operation for this purpose, which SMILE covers through \texttt{SPLIT}.

\textbf{Property Operations.}
This category operates on individual attributes within an entity.
\texttt{ADD}, \texttt{DELETE}, \texttt{RENAME}, and \texttt{COPY} work analogously to their entity-level counterparts.
\texttt{MOVE} transfers an attribute from one entity to another, while \texttt{CAST} changes its data type.
Orion defines \texttt{PROMOTE} and \texttt{DEMOTE} to mark or unmark an attribute as a key, which SMILE covers through \texttt{ADD\_PRIMARY\_KEY}
and \texttt{DELETE\_PRIMARY\_KEY}.

\textbf{Structural Transformations.}
This category is the key enabler for cross-paradigm migration, since each paradigm represents relationships differently.
SMILE defines six operations as three inverse pairs.
\texttt{NEST} and \texttt{UNNEST} convert between references and embedded sub-objects.
\texttt{FLATTEN} and \texttt{UNFLATTEN} promote or group nested fields.
\texttt{WIND} and \texttt{UNWIND} collapse or expand arrays into separate entities.
Of these, \texttt{NEST}, \texttt{FLATTEN}, and \texttt{UNWIND} originate from prior work within our research group, and the remaining three are
their inverses added by SMILE to support bidirectional migration.
The prior \texttt{LINKING} operation is subsumed by \texttt{WIND}.

\textbf{Reference and Embedded Operations.}
These two categories manage relationships at the schema level.
\texttt{ADD\_CONSTRAINT} and \texttt{DELETE\_CONSTRAINT} create
or remove foreign-key references.
\texttt{CAST\_CONSTRAINT} changes the constraint type, and
\texttt{RECARD} modifies the cardinality.
For document-oriented schemas, \texttt{ADD\_EMBEDDED} and
\texttt{DELETE\_EMBEDDED} manage embedded sub-objects.
Orion's morphing operations are handled through \texttt{NEST}
and \texttt{UNNEST}.

\textbf{Key and Constraint Operations.}
This category has no equivalent in any prior approach
(Table~\ref{tab:operation-comparison}).
When migrating into relational targets, SMILE provides
\texttt{ADD\_PRIMARY\_KEY}, \texttt{ADD\_FOREIGN\_KEY}, and
\texttt{ADD\_UNIQUE\_KEY}.
For columnar targets, \texttt{ADD\_PARTITION\_KEY} and
\texttt{ADD\_CLUSTERING\_KEY} are available.
Each \texttt{ADD} has a corresponding \texttt{DELETE} counterpart.

\textbf{Graph-specific Operations.}
Adopted from Hausler et al.~\cite{hausler2023language}, these
operations are required when migrating into or evolving within
the labeled property graph model.
Since the M-Model\textsuperscript{+} represents graph relationship types as entity types with \texttt{entity\_kind=EDGE}, no dedicated relationship type operations are needed---the existing entity type operations handle them directly.
\texttt{ADD\_LABEL} and \texttt{DELETE\_LABEL} handle
Neo4j's multi-label nodes.
\texttt{TRANSFORM} converts a node into a relationship or vice versa.

\subsection{Hub-and-Spoke Architecture}

Supporting all 16 scenarios (Table~\ref{tab:migration-matrix}) through direct point-to-point transformations would require $N \times (N-1) =
O(N^2)$ individual mappings.
To reduce this complexity, the SMILE Engine adopts the Hub-and-Spoke pattern combined with a Canonical Data Model~\cite{hohpe2004enterprise}.
Source schemas are first reverse-engineered into the M-Model\textsuperscript{+} as canonical hub representation, transformed by SMILE scripts,
and then forward-engineered into target schemas, reducing the number of required mappings to $2N = O(N)$.
Figure~\ref{fig:integration-patterns} illustrates the complexity difference.

\subsection{Unified Meta Schema}
\label{sec:unified-meta-schema}

The M-Model\textsuperscript{+} builds on the M-Model~\cite{conrad2022metamodels} with
modifications for cross-paradigm schema migration.
Each database entity is represented as an \texttt{EntityType}
with an \texttt{entity\_kind} that indicates its paradigm
(table, document, graph node, or wide-column family).
SMILE adds a \texttt{labels} field for Neo4j's multi-label
nodes and includes \texttt{is\_key} and \texttt{is\_optional}
flags on each attribute.
Since Neo4j and MongoDB use different key semantics,
\texttt{NODE\_KEY} and \texttt{DOCUMENT\_ID} are added
to the primary key types.
Cassandra's \texttt{tuple<>} columns are covered by
a corresponding \texttt{TupleDataType}.
To support paradigm-specific relationship semantics,
SMILE models connections as \texttt{Reference} (foreign keys),
\texttt{Embedded} (document nesting), or \texttt{Edge}
(graph relationships).

hierarchy, where black elements originate from the M-Model
and blue elements mark SMILE additions.

SMILE is implemented as a prototype in Python~3.13, using
ANTLR4 for parser generation.\footnote{The prototype
implementation is available at
\url{https://github.com/baobaodaren0918/SMILE-3.0}.}
Figure~\ref{fig:system-architecture} shows the system
architecture organized in six layers.

The \emph{interface layer} provides a command-line
interface (\texttt{main.py}) for batch execution
and a REST API (\texttt{web\_server.py}) that serves
a browser-based UI where users can interactively
explore migration and evolution results.

The \emph{core layer} contains the
\texttt{SchemaTransformer}, which implements a handler
for each SMILE operation. Each handler takes an operation
parsed from the SMILE script and applies it to the
M-Model\textsuperscript{+}. The handlers are paradigm-independent
and operate on the same data structures regardless of
the source or target database.

The \emph{parsing layer} uses ANTLR4 to process SMILE
scripts. A \texttt{parser\_factory} detects the grammar
variant (Specific or Generalized) from the file
extension and selects the corresponding grammar.
The \texttt{SMILE\_listeners} module walks the parse tree
and produces an ordered list of operations that the
core layer executes sequentially.

The \emph{hub layer} holds the M-Model\textsuperscript{+}
(Section~\ref{sec:unified-meta-schema}), which serves
as the canonical intermediate representation. All layers
communicate through this shared data model, so no layer
needs to know about any specific database format.

The \emph{adapter layer} provides four database-specific
adapters for PostgreSQL, MongoDB, Neo4j, and Cassandra.
Each adapter handles both reverse engineering (parsing a
source schema into Meta~V1) and forward engineering
(exporting Meta~V2 as target DDL). Because both directions
share the same adapter, adding support for a new database
requires implementing only one module.

The \emph{validation layer} runs two checks after each
migration: a meta-level check that verifies the structural
consistency of Meta~V2, and a round-trip check that
compares the exported target DDL against expected output.

\texttt{config.py} stores all migration and evolution
definitions, which include the source file, the SMILE
script, the source type, and the target type. The core
and adapter layers read these definitions to set up
each run.

\subsection{Pipeline Stages}
\label{sec:pipeline-stages}

Figure~\ref{fig:architecture} illustrates the five-phase pipeline
that transforms a source schema into a target schema.
The pipeline processes two parallel input streams that converge
at a central execution component.
The upper stream carries the source schema through reverse
engineering into the M-Model\textsuperscript{+} (Meta~V1), while
the lower stream parses the SMILE script into an ordered
list of operations.
Both streams feed into the Operation Executor, which applies
the operations to Meta~V1 and produces Meta~V2.
The result is then forward-engineered into the native target
format and delivered to the user through a web-based interface.

\begin{figure*}[t]
  \centering
  \includegraphics[width=\textwidth]{gfx/pipeline1.drawio.pdf}
  \caption{SMILE five-phase pipeline from source schema to target delivery, showing two parallel input streams converging at the Operation Executor.}
  \label{fig:architecture}
\end{figure*}

\textbf{Phase~1 (Input).}
The pipeline begins with two user-provided inputs.
The first is a source schema file in one of four native formats,
namely SQL~DDL for PostgreSQL, JSON~Schema for MongoDB,
Cypher for Neo4j, or CQL for Cassandra.
The second is a SMILE script written in either the Specific
grammar (\texttt{.smel}) or the Generalized grammar
(\texttt{.smel\_gen}), which declares the migration direction
and specifies the desired transformation operations.
These two inputs are independent of each other and are
processed by separate components in the subsequent phase.

\textbf{Phase~2 (Parsing \& Reverse Engineering).}
In this phase, both input streams are processed in parallel
by dedicated components.
On the upper stream, a database-specific source adapter
performs reverse engineering by reading the native schema
file and converting it into the M-Model\textsuperscript{+}.
During this conversion, tables, embedded documents, graph
nodes, and wide-column families are uniformly represented
as \texttt{EntityType} instances, foreign keys become
\texttt{Reference} objects, embedded sub-documents become
\texttt{Embedded} objects, and graph relationships become
\texttt{Edge} objects.
Native data types such as \texttt{VARCHAR}, \texttt{bsonType},
or \texttt{text} are mapped to a shared \texttt{PrimitiveType}
enumeration.
The result of this step is Meta~V1, which serves as the
canonical representation of the source schema.

On the lower stream, the parser factory inspects the file
extension to determine which grammar variant to use and
instantiates the corresponding ANTLR4-generated lexer
and parser.
A custom listener walks the resulting parse tree and emits
an ordered list of \texttt{Operation} objects, each carrying
an \texttt{OpType} enum value and a parameter dictionary
that encodes the operation's arguments.
As shown in the grammar variants box in
Figure~\ref{fig:architecture}, the Specific grammar uses
dedicated compound keywords such as \texttt{ADD\_ATTRIBUTE}
and \texttt{RENAME\_ENTITY}, while the Generalized grammar
combines a verb with an object keyword such as
\texttt{ADD ATTRIBUTE} and \texttt{RENAME ENTITY}.
Both variants produce semantically identical operation lists.

\textbf{Phase~3 (Execution).}
The two streams converge at the Operation Executor, which
receives Meta~V1 from the upper stream and the parsed
operation list from the lower stream.
The executor iterates over the operations in their original
order and dispatches each one to a dedicated handler method.
These handlers cover the full operation taxonomy described
in Section~\ref{sec:operation-taxonomy}, including entity
operations, attribute operations, structural transformations
such as \texttt{NEST} and \texttt{FLATTEN}, key and constraint
operations, and graph-specific operations.
Each handler modifies the in-memory Meta Schema directly,
and the executor tracks all changes for later reporting.
After all operations have been applied, the executor normalizes
entity kinds to match the target paradigm.
For instance, in a \texttt{FROM RELATIONAL TO DOCUMENT}
migration, all \texttt{TABLE} entity kinds are converted to
\texttt{DOCUMENT}.
The resulting data structure is Meta~V2, which represents
the transformed schema in the canonical format.

\textbf{Phase~4 (Forward Engineering \& Output).}
The target adapter reads Meta~V2 and exports it into the
corresponding native schema definition language.
Since each adapter module implements both a \texttt{parse}
method for reverse engineering and an \texttt{export} method
for forward engineering, the same adapter that was used as
a source adapter in Phase~2 can serve as the target adapter
in this phase.
For relational targets, the adapter generates PostgreSQL
\texttt{CREATE TABLE} statements with primary keys, foreign
keys, and unique constraints.
For document targets, it produces a MongoDB JSON~Schema
with nested \texttt{properties} and \texttt{bsonType}
annotations.
For graph targets, it emits Cypher node and relationship
type constraints.
For columnar targets, it outputs Cassandra \texttt{CREATE TABLE}
statements with partition and clustering key definitions.

An optional two-layer validation can be executed after the
export step.
Layer~1 compares Meta~V2 against the expected target schema
that is obtained by parsing the ground-truth target file,
thereby verifying that the SMILE script produces the correct
structural result.
Layer~2 performs a round-trip check by parsing the exported
target DDL back into a meta representation through the
adapter's reverse engineering and comparing it against the
same expected schema, thereby verifying that the adapter's
forward engineering output is consistent and complete.

\textbf{Phase~5 (Delivery).}
The exported target schema, together with the operation
execution log and the validation results, is delivered
to the user through a browser-based Web~UI.
This interface visualizes the complete migration process,
displaying the source schema (Meta~V1) and the result
schema (Meta~V2) side by side, listing each executed
operation with its status, and presenting the generated
target DDL in syntax-highlighted form.
The validation outcomes are shown as a summary indicating
whether both layers passed, along with detailed diagnostics
for any discrepancies.

\section{Conclusion}
\label{sec:conclusion}
% ============================================================

We presented SMILE, a domain-specific language that unifies
schema migration (inter-system) and schema evolution (intra-system)
within a single ANTLR4-based grammar.
SMILE builds on the M-Model~\cite{conrad2022metamodels},
integrates evolution operations from the Orion
taxonomy~\cite{chillon2022taxonomy} with structural
transformations from prior work within our research group,
and extends them with key/constraint and graph-specific operations.
A prototype with adapters for PostgreSQL, MongoDB, Neo4j,
and Cassandra validates the approach across all 16~migration
and evolution directions, with 32/32~configurations passing
the two-layer validation pipeline.

\textbf{Limitations.}
First, SMILE addresses schema migration only and does not
handle data migration.
In enterprise environments such as SAP, schema structures
are tightly coupled with business logic, stored procedures,
access control, and cross-module dependencies
(e.g., between MM, FI, and SD modules),
making schema-level transformations alone insufficient
for a complete migration.
Second, the MongoDB adapter assumes a single root document
per collection, which does not account for structural
variations where documents within the same collection
follow different schemas---a common pattern in
schema-on-read environments.
Third, SMILE performs one-shot transformations without
schema version tracking.
Unlike Orion~\cite{chillon2022taxonomy}, which maintains
a history of schema changes, SMILE does not support
incremental evolution or multi-version coexistence.
Fourth, the 16~nullability notices observed in the
evaluation indicate that cross-paradigm semantic
differences (e.g., Cassandra clustering keys are
inherently non-nullable) are not fully resolved
automatically and require manual review.

\textbf{Future work.}
A natural next step is to extend SMILE with data migration
capabilities, generating ETL scripts that complement
the schema transformations.
Integrating schema version management, similar to
Darwin's~\cite{storl2022darwin} version tracking,
would enable incremental evolution and historical
schema auditing.
Finally, developing Language Server Protocol support
for SMILE would provide syntax highlighting,
auto-completion, and real-time validation in
modern IDEs, lowering the barrier to adoption.
